# MLOps End-to-End: Credit Underwriting + Fraud Detection

A complete MLOps reference project on **Databricks** for a Lending Fintech. Two models are trained, governed, deployed, monitored, and retrained — entirely on Databricks-native features. All scripts are **workspace-agnostic**: catalog, schema, model names, and endpoints are read from a single config so the same code runs in any workspace.

## Data model

Six raw tables, **1000 rows each**, generated by `scripts/generate_synthetic_data.py`:

1. **applications** — loan applications with amounts, terms, declared income, channel
2. **kyc_identity** — customer identity (PII-style with fake data)
3. **credit_bureau** — bureau scores, utilization, inquiries, tradelines
4. **internal_transactions** — checking/savings deposits, withdrawals, repayments
5. **fraud_events** — verification flags, device fingerprint anomalies, watchlist hits
6. **historical_outcomes** — labels: defaulted (Y/N), fraud (Y/N) for already-decisioned apps

The two business decisions are:

- **Credit risk** — will the applicant default in the next 12 months?
- **Fraud risk** — is this application fraudulent?

## Before you run

Three prerequisites in your target workspace:

1. **A Unity Catalog exists that you can write to.** Set its name in `conf/config.yaml` *and* in `databricks.yml` (or pass `--var catalog=<name>` on deploy). Some workspaces don't allow `CREATE CATALOG IF NOT EXISTS` from SQL — pre-create the catalog via the Catalog Explorer if needed.
2. **You have `CAN_USE` on the catalog and `CAN_CREATE` on the schema** (or `ALL_PRIVILEGES`). The pipeline creates schemas, volumes, tables, MLflow experiments, registered models, and serving endpoints. The bootstrap notebook will fail fast with a clear error if a permission is missing.
3. **Your IP is allow-listed if the workspace has IP ACLs enabled.** Otherwise CLI / DAB commands will fail with `IP blocked`.

## Quickstart

```bash
# 1. Edit conf/config.yaml — replace <change_me_to_your_catalog> with your UC catalog
# 2. Edit databricks.yml — set the dev/prod workspace host (or pass via DATABRICKS_HOST)
# 3. Authenticate
databricks auth login https://<your-workspace> --profile=<profile>

# 4. Deploy the bundle (dev target)
databricks bundle deploy --profile=<profile> --target=dev

# 5. Run the end-to-end job
databricks bundle run mlops_end_to_end_pipeline --profile=<profile> --target=dev
```

## Configuration (workspace-agnostic)

All paths are derived from `conf/config.yaml` → loaded by `notebooks/00_setup/config.py`. To run in another workspace, you only change values in that one file (or override via bundle variables).

```yaml
catalog: <change_me_to_your_catalog>
schema: credit_fraud
volume: raw_landing
credit_model_name: credit_risk_model
fraud_model_name: fraud_detection_model
credit_endpoint_name: credit-risk-endpoint
fraud_endpoint_name: fraud-detection-endpoint
```

## CI/CD

- PR opened → GitHub Actions validates the bundle (`databricks bundle validate`) and deploys to the `dev` target.
- Merge to `main` → deploys to the `prod` target and runs the smoke test.
- Drift detected → `retrain` job triggers, registers a new candidate, runs validation, promotes via alias only if it beats the current `@champion`.

See `.github/workflows/ci.yml`.
